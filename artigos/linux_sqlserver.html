SQL Server no Linux: Guia inicial

Esse artigo descreve um passo fundamental para o DBA SQL Server: a adaptação ao ambiente Linux. Seja para baratear custos de licenciamento de sistema operacional, padronizar pipelines com contêineres ou unificar a stack de servidores, entender como gerenciar o motor do SQL Server fora do seu habitat natural é vital.

Sabendo disso, vamos começar pelo choque de realidade: sair do conforto do Windows Server.

No ecossistema do PostgreSQL, o Linux é a casa nativa. DBAs de Postgres estão acostumados a gerenciar processos, analisar consumo de memória e fazer o troubleshooting de I/O diretamente pelo terminal. Para os DBAs de SQL Server, que historicamente dependem do Task Manager, do PerfMon e do SSMS para tudo, a tela preta pode parecer hostil. Mas os conceitos são os mesmos, apenas as ferramentas mudam.

Quando o servidor apresenta lentidão, a primeira reação no Windows é abrir o Task Manager. No Linux, nós acessamos o terminal e utilizamos ferramentas nativas que leem diretamente as informações do kernel.

1. CPU e Memória (O substituto do Task Manager)

Para monitoramento em tempo real de processos, o comando clássico é o top (ou htop, sua versão aprimorada e interativa). Ele entrega a visão do consumo de CPU por núcleo, uso de memória RAM e Swap. Para o SQL Server, você vai procurar pelo processo principal chamado sqlservr.

# Abre o monitor de processos interativo. Pressione 'F4' para filtrar por 'sqlservr' e 'q' para sair.
htop

Se você precisa de uma visão rápida e estática apenas do consumo global de memória RAM (total, usada, livre e em cache/buffer), o comando free é a escolha certa:

# Exibe o uso de memória do sistema em formato legível para humanos (Megabytes/Gigabytes)
free -h

2. Armazenamento e I/O (O substituto do Windows Explorer e Resource Monitor)

Diferente do Windows com seus discos C:, D: e E:, o Linux usa uma árvore de diretórios unificada montada na raiz (/). Para verificar o espaço livre em disco, usamos o df -h. Para saber o tamanho de uma pasta específica (como os dados do banco), usamos o du -sh.

# Verifica o espaço livre em todas as partições montadas
df -h

# Verifica o tamanho total do diretório de dados padrão do SQL Server
sudo du -sh /var/opt/mssql/data/

Mas para um DBA, espaço livre não basta; precisamos ver o gargalo de leitura e escrita (I/O). O iostat é fundamental para medir as IOPS e a latência de disco no momento de lentidão:

# Exibe estatísticas estendidas de I/O em disco, atualizando a cada 2 segundos
iostat -xd 2
Fato curioso: A Microsoft manteve a estrutura lógica tão fiel que os arquivos de dados continuam sendo .mdf, .ndf e .ldf, mesmo rodando em um sistema de arquivos ext4 ou XFS no Linux.

3. Serviços e Logs do SO (O substituto do services.msc e Event Viewer)

No Linux moderno, quem gerencia os serviços rodando em background (daemons) é o systemd. Em vez de abrir painéis gráficos para reiniciar o serviço do banco, tudo é feito via systemctl. E para ler os logs registrados pelo SO (caso o serviço tenha "crachado" antes mesmo de iniciar o SQL), usamos o journalctl.

# Verifica se o SQL Server está rodando, se falhou ou se está inativo
sudo systemctl status mssql-server

# Para reiniciar o serviço do SQL Server brutalmente
sudo systemctl restart mssql-server

# Lê os últimos 50 logs de sistema gerados especificamente pelo SQL Server
sudo journalctl -u mssql-server -n 50

4. Lendo Logs Internos e Arquivos (O substituto do Notepad / SSMS Log Viewer)

No Windows, você abriria o SSMS ou o Bloco de Notas para analisar o famoso ERRORLOG do SQL Server. No Linux, nós lemos arquivos de texto gigantescos instantaneamente pelo terminal sem travamentos. O tail lê o final do arquivo, enquanto o grep atua como um "Ctrl+F", buscando trechos específicos.

# Exibe as últimas linhas do ERRORLOG em tempo real (acompanhando a gravação ao vivo)
tail -f /var/opt/mssql/log/errorlog

# Busca exclusivamente por falhas de login (Login failed) dentro do ERRORLOG
grep "Login failed" /var/opt/mssql/log/errorlog

5. Rede e Conectividade (O substituto do netstat do Windows)

A aplicação não consegue conectar? Antes de culpar o banco, você precisa verificar se o SQL Server realmente subiu e se está escutando na porta correta (1433 por padrão). O comando ss (Socket Statistics) substitui o netstat.

# Lista as portas TCP abertas (Listening), numéricas, e mostra qual processo as está usando
sudo ss -tlnp | grep sqlservr

Como funciona o SQL Server no Linux por baixo dos panos?

Aqui precisamos minar um ponto crucial. Diferente do PostgreSQL, que foi escrito com chamadas diretas às APIs nativas do kernel Linux (POSIX), o motor relacional do SQL Server possui milhões de linhas de código fortemente atreladas às APIs do Windows (Win32).

A Microsoft não reescreveu o SQL Server do zero. Ela criou uma camada de abstração brilhante chamada SQLPAL (SQL Server Platform Abstraction Layer). Baseada no antigo projeto de pesquisa Drawbridge, o SQLPAL atua como um "mini sistema operacional" embutido dentro do processo do Linux. Ele traduz em tempo real as chamadas do motor do banco de dados para chamadas que o kernel do Linux consegue entender.

  • Compatibilidade total: Devido ao SQLPAL, o motor continua utilizando as mesmas estruturas de agendamento (SQLOS). O banco "acha" que está no Windows.
  • Migração transparente: É por conta dessa arquitetura que você pode fazer um backup (.bak) em um Windows Server 2022 e restaurar perfeitamente em um servidor Linux.

Instalação e Ferramentas: Sem interface gráfica

Se você usou Linux na vida, aqui vai a maior diferença conceitual: no Windows, você entra no site da Microsoft, baixa um arquivo .iso de gigabytes, monta a imagem e roda um setup.exe. No Linux, nós usamos Gerenciadores de Pacotes (como o apt no Ubuntu ou dnf no RedHat).

Em vez de ir atrás do arquivo, você diz ao sistema operacional através do terminal onde estão os servidores da Microsoft, e o próprio Linux baixa, descompacta e instala tudo no lugar certo. Isso facilita a vida de quem trabalha com automação (Ansible, Terraform) e Infraestrutura como Código.

Usando o Ubuntu como exemplo (a distribuição mais amigável para iniciantes), o processo físico se resume a estes comandos:

# 1. Importa a chave de segurança da Microsoft (para garantir que o download é oficial)
curl -fsSL https://packages.microsoft.com/keys/microsoft.asc | sudo gpg --dearmor -o /usr/share/keyrings/microsoft-prod.gpg

# 2. Adiciona o repositório do SQL Server 2022 na "lista telefônica" do seu Linux
curl -fsSL https://packages.microsoft.com/config/ubuntu/22.04/mssql-server-2022.list | sudo tee /etc/apt/sources.list.d/mssql-server-2022.list

# 3. Atualiza a lista de pacotes disponíveis e instala o motor do banco de dados
sudo apt-get update
sudo apt-get install -y mssql-server

Pronto, os binários estão instalados. Mas o banco ainda não está rodando. Falta a configuração final (collation, portas e a famosa senha do SysAdmin). Para isso, a Microsoft criou um utilitário exclusivo chamado mssql-conf.

# Executa o assistente de configuração inicial interativo
sudo /opt/mssql/bin/mssql-conf setup

Ao rodar o comando acima, o terminal fará duas perguntas cruciais para você:

  1. A Edição do SQL Server: Ele vai listar opções de 1 a 8 (Evaluation, Developer, Express, Standard, Enterprise...). Digite o número correspondente à sua necessidade e Enter.
  2. A senha do sa: Aqui vai um detalhe que todo iniciante no Linux acaba se confundindo: por questões de segurança, ao digitar senhas no terminal, nada aparece na tela. Nem mesmo asteriscos. O Linux está capturando as teclas. Apenas digite sua senha forte e aperte Enter.

E quanto à administração no dia a dia?

O SSMS é maravilhoso, mas é exclusivo do Windows. Para um fluxo de trabalho multiplataforma, temos duas opções principais:

  • Azure Data Studio (ADS): É a recomendação moderna. Construída sobre o Electron (mesma base do VS Code), roda nativamente no Linux. Possui suporte excelente para T-SQL, visualização de planos de execução gráficos e extensões poderosas.
  • DBeaver: Baseado em Java, é um cliente universal extremamente popular na comunidade Linux. Se você gerencia múltiplos motores de banco de dados no seu dia a dia, ele se conecta perfeitamente ao SQL Server (via driver JDBC) e oferece uma interface gráfica robusta e familiar que lembra bastante a flexibilidade estrutural do SSMS.
  • sqlcmd: A ferramenta oficial de linha de comando. Essencial para quando você está conectado via SSH em um servidor sem interface gráfica e precisa executar consultas rápidas ou scripts de manutenção.
# Conectando localmente via sqlcmd para testar a instância
sqlcmd -S localhost -U sa -P 'SuaSenhaForte123!' -Q "SELECT @@VERSION"

A curva de aprendizado inicial pode assustar quem nunca saiu do ecossistema Microsoft, mas os fundamentos do motor de banco de dados permanecem os mesmos. Dominar a CLI do Linux não apenas expande seu portfólio de administração e troubleshooting, mas é o primeiro passo para orquestrar bancos de dados robustos em arquiteturas orientadas a contêineres.
