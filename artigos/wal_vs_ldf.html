WAL x .ldf

Esse artigo servirá de guia prático e teórico em relação ao gerenciamento do log nos sgbd's e postgresql. Que apesar de soar semelhante, seguem bases completamente diferentes.

Sabendo disso, vamos começar pelo SQL Server e os arquivos .ldf:

Apesar do PostgreSQL chamar o arquivo fisicamente de WAL, esse é o nome dado ao conceito, que também é utilizado nos arquivos de log (.ldf). WAL, ou Write-Ahead Logging, siginifica que qualquer modificação de dados ou estrutura é gravada primeiramente no log de transação do disco, antes mesmo de qualquer página de dados ser gravada no arquuivo de dados (.mdf). Isso serve, também, para casos onde há uma queda inesperada do banco, o SQL Server lê o log de transação na inicialização e refaz (redo) ou desfaz (undo) as transações, visando cumprir com o "C" do conceito ACID, ou seja, conscistência.

Seguindo a premissa do mecanismo do banco de dados, temos algumas especificações no SQL Server, como os VLF's.

VLF ou Virtual Log Files, teórica e resumidamente, se trata de partições lógicas dentro do arquivo de log, responsáveis pelo controle transacional a um nível mais granular. Funciona de forma circular, preenchendo o VLF 1, VLF 2, VLF 3 e assim for diante, até o último. Após o último, se o primeiro já estiver inativo (transação consolidada), ele sobreescreve o VLF 1.

Fato curioso: Uma má gestão do autogrowth, pode gerar fragmentação dos VLF's, criando milhares de VLF's por arquivo, destruindo a performance de recuperação (recovery) e degradando operações de escrita.

Além dessas partições lógicas, possuímos os modelos de recuperação (recovery model):

O recovery model gerencia e define como o crescimento do log irá acontecer, além de outros conceitos relacionados a HADR. Para explicar do que se trata, nada melhor do que citar as principais opções (as comumente utilizadas):

  • SIMPLE: Transações finalizadas são marcadas como inativas após um CHECKPOINT. O log é truncado (termo utilizado para simbolizar que acontece um comando TRUNCATE) e reutiliza o espaço. Faz com que o log não cresça indefinidamente, mas perde a possibilidade de um recovery point-in-time.
  • FULL: Absolutamente tudo é logado. Um VLF só é marcado para ser reutilizados após um backup de log. Se ninguém realizar o backup de log, o ciclo circular é quebrado, e assim pode chegar a gerar o temido erro 9002, causando a parada do banco por lotar o disco.

Além disso, possuímos um conceito primordial: O LSN (Log Sequence Number) que, de forma bem direta, funciona como um localizador para o seu banco de dados.

Independentemente do motor, todo registro que entra no log ganha esse número de série. É isso que garante que as transações sejam reproduzidas exatamente na ordem em que aconteceram em caso de uma falha.

No SQL Server, se parace com isso: 00000032:0000012a:0001

Trẽs blocos que significam: Id do VLF, deslocamento dentro do bloco do log e slot da transação.

O conceito por traz dos arquivos de log é complexo e cheio de casos interessantes. Que será citado em outros artigos focados 100% nesses casos, já que os mesmos, por si só, são assuntos longos e ricos em conteúdo.

após o overview relacionado aos arquivos de log do SQL Server, podemos partir para WAL do sistema de gerenciamento de banco de dados: PostgreSQL.

Como dito no início, WAL é o nome dado ao conceito, mas no postgres, também nomeia o arquivo. Ou melhor, o diretório dos arquivos.

Vamos passar primeiramente pela arquitetura física:

Enquanto o SQL Server gerencia apenas um único arquivo (exemplo.ldf) e o divide logicamente em partições. O postgres possui um diretório responsável por isso, chamado pg_wal. (ou, em versões antigas, pg_xlog.)

Dentro do diretório, o postgres cria diversos arquivos físicos com nomenclatura padrão hexadecimal (000000010000000000000001). Por padrão, cada arquivo desse possui 16MB e é chamado de Segmento de WAL.

O ciclo circular dos VLF's do SQL Server ocorrem aqui quase da mesma maneira, mudando apenas o fato de que não se trata de partições de um único arquivo, e sim arquivos de um diretório. Ou seja, com o decorrer das gravações das transações, os arquivos são reciclados e renomeados.

Recovery model no postgres e seu paradigma:

Aqui, não temos FULL ou SIMPLE, e isso é o que o DBA SQL Server precisa entender para que não caia no susto de tentar alterar o recovery model da base pela interface gráfica, pois o gerenciamento desse recurso, assim como da maioria, é feito de forma global através do arquivo de configuração: postgresql.conf.

wal_level: Define o quanto de informação vai ser escrita no WAL, podendo ser minimal, replica e logical:

  • minimal: O mínimo para o banco não corromper em um crash.
  • replica: O suficiente para Point-in-time Recovery e replicação física.
  • logical: O suficiente para replicação lógica.

O equivalente ao FULL do SQL Server, você precisa ativar o archive_mode, isso faz com que o postgres execute um script (o archive_command) toda vez que um arquivo de log (16MB) encha, o copiando para um storage externo. Pois, caso contrário, estando ali, seria reciclado, cedo ou tarde.

O LSN (Logical Sequencial Number) no postgres não é só uma sequẽncia lógica, mas soa mais como um ponteiro de posição física (byte offset de 64 bits) e marca exatamente qual byte do arquivo a transação foi gravada.

Se parece com isso: 16/B374D848

Além disso o postgres tem um tipo de dado nativo, apenas para lidar com isso, o pg_lsn.

O comando CHECKPOINT foi citado anteriormente no recovery SIMPLE do SQL Server, mas no postgres ele também acontece, e é um evento crucial para a saúde do WAL. Pois o mesmo, é responsável por definir quando as páginas em memória (RAM, Shared Buffers, Cache, etc.) e as passa para o disco. O que, de maneira mais demonstratativa, ocorre como um caixa de supermercado, onde o CHECKPOINT é o operador de caixa, os arquivos seriam os clientes e as compras sendo os dados.

Ok, sabendo disso, o que pode levar o WAL a afetar o disco? No SQL Server, citamos alguns casos onde o disco pode ser afetado por falta de algumas configurações. Aqui, o que pode fazer o disco ser afetado, é o mal uso das configurações aplicadas. Sendo duas delas:

  • Falha no archiving: Caso o archive_command não funcione, ao contrário do que você pode estar pensando, o postgres não deleta ou perde os arquivos, eles o mantém localmente. E como nesse caso não há vazão, os log seguem se acumulando, o que om excesso de erros, pode afetar o disco.
  • wal_keep_size muito alto: Em resumo, esse parâmetro define o quanto em GB de WAL o banco deve guardar por segurança. Mal configurado, pode ocupar bastante espaço e até mesmo topar.

Há exemplos relacionados a replicação que envolver o WAL, não só no postgres, mas o conteúdo abrange diversos temas complexos relacionados a recuperação de desastres e alta disponibilidade.

Para finalizar, o que importa além de tudo para nós DBA's, as queries. Começando pelo SQL Server:

Se quiser verificar o tamanho do .ldf de cada base e o percentual de uso, você pode usar o:

DBCC SQLPERF(LOGSPACE);

Já essa query explica quem pode ser o ofensor do aumento desenfreado do log:

SELECT
name AS DatabaseName,
recovery_model_desc AS RecoveryModel,
log_reuse_wait_desc AS MotivoRetencaoLog
FROM sys.databases
WHERE database_id > 4; -- Ignora as bases de sistema

Dica: Se o resultado for LOG_BACKUP, o banco está em modelo FULL esperando o backup de log. Se for ACTIVE_TRANSACTION, alguém abriu uma transação, mas ainda não ocorreu o COMMIT. Além de outros reuse_waits.

Se quiser monitorar os VLF's, saber quantos tem no banco atual, para entender o nível de fragmentação interna (e caso o seu servidor possua SQL Server 2017 ou superior), use isso:

SELECT
DB_NAME(database_id) AS DatabaseName,
COUNT(vlf_sequence_number) AS Quantidade_VLFs
FROM sys.dm_db_log_info(DB_ID())
GROUP BY database_id;

Resumidamente e de forma básica, isso servirá para suas primeiras impressões. Agora podemos partir para o postgres:

Se sua versão do postgres for 10+, você não precisar verificar o diretório da pasta de logs via terminal, basta rodar isto e listar:

SELECT
COUNT(*) AS quantidade_arquivos_wal,
SUM(size) / 1024 / 1024 AS tamanho_total_mb
FROM pg_ls_waldir();

Para descobrir o segmento de WAL atual, no exato milisegundo:

SELECT pg_walfile_name(pg_current_wal_lsn()) AS arquivo_wal_atual;

Como o CHECKPOINT só limpa o WAL do que já foi consolidado, longas transações podem segurar o WAL e inflar o disco, para verificá-las:

SELECT
pid,
usename AS usuario,
client_addr AS ip_origem,
now() - xact_start AS tempo_transacao,
query AS comando_executado
FROM pg_stat_activity
WHERE state = 'active'
AND xact_start < now() - INTERVAL '5 minutes'
ORDER BY tempo_transacao DESC;

Essas são as principais diferenças e semelhanças entre o conceito de arquivo de log no PostgreSQL x SQL Server. Mais detalhes sobre temas citados aqui, serão citados em outros artigos. Obrigado por ler até aqui e bons estudos!
