E-Commerce Analytics Pipeline

Orquestração de pipeline de dados automatizado utilizando uma Modern Data Stack rodando inteiramente em containers.

Este projeto consiste em um pipeline end-to-end de engenharia de dados que simula o fluxo completo de uma operação de e-commerce. A arquitetura foi desenvolvida para orquestrar desde a geração e ingestão de dados brutos (simulando registros "sujos") até a entrega de métricas limpas e modeladas em um banco de dados relacional (Serving Layer).

Fluxo de Orquestração (Airflow DAG)

O pipeline é executado em um cronograma diário totalmente automatizado pelo Apache Airflow, seguindo as etapas:

  1. Ingestão (Raw): Geração de dados sintéticos de e-commerce (pedidos, eventos, produtos) e upload direto para o MinIO (Data Lake local compatível com S3).
  2. Processamento e Limpeza: Rotinas em PySpark leem os arquivos brutos, aplicam validações de schema e limpam dados intencionalmente corrompidos.
  3. Enriquecimento e Agregação: O Spark cruza (JOIN) as tabelas de pedidos com informações de produtos e calcula KPIs primários (ex: faturamento diário, top produtos e retenção).
  4. Carga no Data Warehouse: Os resultados agregados são ingeridos em um banco PostgreSQL.
  5. Modelagem Dimensional: O dbt (Data Build Tool) assume a transformação final, construindo as camadas staging e marts para disponibilizar as tabelas finais para consumo analítico.

Stack Tecnológica

  • Apache Spark (PySpark): Limpeza de dados, enriquecimento e agregações pesadas.
  • Apache Airflow: Agendamento, orquestração e paralelismo das tarefas.
  • dbt (Data Build Tool): Transformação analítica e documentação no Data Warehouse.
  • PostgreSQL: Camada de consumo (Serving Layer) para as tabelas finais.
  • MinIO: Armazenamento de objetos (Local S3-compatible).
  • Docker Compose: Provisionamento e isolamento de toda a infraestrutura local.

DataOps: Execução e Deploy

A infraestrutura é provisionada via Docker, garantindo reprodutibilidade. Abaixo estão os passos para inicializar a stack.

1. Crie os diretórios de persistência e ajuste permissões do Airflow:

mkdir -p logs/scheduler dags plugins data
chmod -R 777 logs data dbt_project
echo "AIRFLOW_UID=$(id -u)" >> .env

2. Faça o download dos drivers (JARs) necessários para o Spark comunicar com o MinIO (S3) e PostgreSQL:

mkdir -p spark/jars
wget https://repo1.maven.org/maven2/org/apache/hadoop/hadoop-aws/3.3.2/hadoop-aws-3.3.2.jar -P spark/jars/
wget https://repo1.maven.org/maven2/com/amazonaws/aws-java-sdk-bundle/1.11.901/aws-java-sdk-bundle-1.11.901.jar -P spark/jars/
wget https://repo1.maven.org/maven2/org/postgresql/postgresql/42.7.3/postgresql-42.7.3.jar -P spark/jars/

3. Construa e inicie os containers (Acesse a interface do Airflow via localhost:8080 e ative a DAG):

docker compose build
docker compose up airflow-init
docker compose up -d

Preparação para Nuvem (Cloud Readiness)

O pipeline foi projetado visando portabilidade para ambientes cloud. Para migrar para a AWS, as alterações são mínimas e configuráveis via variáveis de ambiente, não exigindo reescrita no código PySpark:

  • Substituir a URI do MINIO_ENDPOINT pelo endpoint da região do AWS S3.
  • Injetar as Access Keys do IAM da AWS.
  • Substituir a conexão local do PostgreSQL pelo endpoint de um banco AWS RDS.
